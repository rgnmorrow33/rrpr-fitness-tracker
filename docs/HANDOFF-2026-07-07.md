### 2. Pre-push syntax gate - `scripts/hooks/pre-push-syntax-check.js`

Why: "always run node --check before push" had no backstop, and CI
(smoke.yml) runs post-deploy, so a syntax error would reach Selisa's iPads
~30 seconds after a bad push.

What it does: a Claude Code PreToolUse hook. When a Bash/PowerShell command
containing `git push` is about to run, it extracts every inline `<script>`
block from `RoundRock_Fitness_Tracker.html` **as of HEAD** (the content the
push would ship, not the working tree), pads line numbers so errors report
real HTML line numbers, and runs `node --check`. Failure blocks the push
(hook exit 2) and feeds the syntax error back to Claude. It also blocks if
the tracker file is missing/renamed at HEAD, which incidentally enforces the
"do NOT rename the tracker file" rule. Unexpected hook errors fail OPEN
(exit 1) so a bug in the guard can never freeze all pushes.

### 3. Pre-push tag gate - `scripts/hooks/pre-push-tag-check.js`

Why: the tag rule lived only in CLAUDE.md prose and demonstrably failed
(v4.35).

What it does: same PreToolUse shape. On `git push`, it scans commit messages
in the outgoing range (`origin/<branch>..HEAD`, falling back to
`origin/main..HEAD` for new branches) for `vX.Y` references and blocks if
any referenced version has no matching git tag, printing the exact commands
to fix it.

**Convention shift:** this enforces tag-BEFORE-push. CLAUDE.md currently
says tag AFTER the push. The tag still marks the exact commit that goes
live; the order just flips (tag locally, push branch, push tag). CLAUDE.md's
"Version tagging" section needs a wording update - deferred to the CLAUDE.md
diet session.

### 4. Hook wiring tracked in-repo - `.claude/settings.json` + `.gitignore`

Why: the entire setup was device-local; if this machine dies the enforcement
dies with it.

`.claude/settings.json` (now tracked) registers both hooks under
`PreToolUse` with matcher `Bash|PowerShell`. `.gitignore` changed from

## Known limitations / activation notes

- Hooks registered in settings load at session start: they become active
  the NEXT Claude Code session (review them in the /hooks menu). The scripts
  themselves are fully tested above via the real stdin contract.
- These are Claude Code hooks: they gate pushes made THROUGH Claude Code.
  A manual `git push` from a plain terminal bypasses them. If that matters,
  a follow-up can add a tracked githooks/ dir + `git config core.hooksPath`
  for git-level enforcement on any machine.
- Hook commands use paths relative to the project root (the normal hook
  cwd). Both scripts also self-resolve the repo root via CLAUDE_PROJECT_DIR
  / `git rev-parse` for the files they inspect.
- The tag hook only reads commits on the current branch's outgoing range;
  it does not parse push refspecs (pushing a ref other than HEAD is not this
  repo's flow).
